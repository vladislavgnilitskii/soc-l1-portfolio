# 🧠 Система обучения — SOC L1

> Как учиться так, чтобы знания оставались, а не «прочитал и забыл». Кладётся в репозиторий: `notes/learning-system.md`

---

## Шесть принципов (на них держится всё)

**1. Первоисточник — до меня, не вместо.** Сначала сам читаешь документ/статью/лабу. Потом приходишь с вопросами. Если наоборот — я становлюсь костылём, и получается ровно та «иллюзия компетентности», которая уже была с Go-кодом. Чат хорош для разбора непонятного и для проверки тебя, но он плохой единственный источник: я даю сжатую выжимку, а на собесе спрашивают детали.

**2. Конспект пишется по памяти, при закрытом материале.** Прочитал → закрыл всё → написал своими словами → только потом сверил и дописал красным, что упустил. Это retrieval practice — единственное, что реально строит память. Переписывание с экрана бесполезно.

**3. Конспект строится вопросами, а не темами.** На собесе не спросят «расскажи про Windows-логи», спросят «чем 4624 отличается от 4625». Конспект в формате вопросов сразу тренирует нужный формат извлечения.

**4. Практика руками — каждый день, даже в теоретическую неделю.** Нет практики → нет навыка. В неделю 1 практика = разбор кейсов и команды в терминале, дальше = лаборатория.

**5. Интервальные повторения: 1 → 3 → 7 → 14 дней.** Каждая новая тема повторяется по карточкам на следующий день, через 3, через 7 и через 14. Без этого неделя 2 сотрёт неделю 1.

**6. Тест Фейнмана: объясни вслух простыми словами.** Если не можешь объяснить без терминов-подпорок — не понял. Проговаривать именно **вслух**, а не «в голове»: на собесе провал случается не в знании, а в артикуляции.

## Структура учебного дня (3 ч 20 мин)

|Блок|Время|Что делаешь|
|---|---|---|
|**1. Разгон**|20 мин|Карточки по расписанию повторений. Без подглядывания. Что не вспомнил — помечаешь|
|**2. Вход в тему**|60 мин|Читаешь первоисточник САМ. Выписываешь непонятное списком|
|_перерыв_|10 мин|Встать, отойти от экрана. Не телефон|
|**3. Разбор**|30 мин|Приходишь ко мне со списком непонятного. Спрашиваешь, спорим, уточняем|
|**4. Практика**|60 мин|Руками: команды, лаба, разбор кейсов, триаж-сценарии|
|**5. Конспект**|30 мин|Материал ЗАКРЫТ. Пишешь по памяти в формате вопросов. Потом сверяешь|
|**6. Закрепление**|20 мин|Делаешь 5–10 карточек по теме дня + коммит в git|

**Итого ~3 ч 20 мин.** Если есть 4 часа — добавляй в блок 4 (практика), не в чтение.

## Карточки и повторения

### Вариант А — Anki (рекомендую)

Установка на Arch: `sudo pacman -S anki` Колода: `SOC L1`. Подколоды: `Process`, `Windows`, `Linux`, `Network`, `Detection`, `Tools`.

Формат карточки — строго вопрос→ответ, коротко:

- Лицевая: `Logon Type 10 — это что?`
- Оборотная: `RemoteInteractive = RDP. Отличать от Type 3 (сетевой, SMB)`

Anki сам считает интервалы — это его главная ценность.

### Вариант Б — plain markdown (если не хочешь ставить Anki)

Файл `notes/flashcards.md`:

```markdown
| Вопрос | Ответ | Тема | Дата | 1д | 3д | 7д | 14д |
|---|---|---|---|---|---|---|---|
| Что значит 4625? | Неудачный вход (не выход!) | Windows | 23.07 | ✅ | ✅ | ⬜ | ⬜ |
| Logon Type 3 vs 10? | 3 — сетевой (SMB); 10 — RDP | Windows | 23.07 | ✅ | ⬜ | ⬜ | ⬜ |
```

Хуже Anki (сам считаешь даты), но не требует ничего ставить и коммитится в git как есть.

### Правило карточек

- 5–10 карточек в день, не больше. Больше — не выучишь, будет отвращение.
- Одна карточка = один факт. Не «расскажи про Kerberos», а «какой Event ID при запросе TGS?»
- Карточка делается **сразу после конспекта**, пока тема свежая.

```bash
git add .

```

Один день = один коммит с осмысленным сообщением. К концу роадмапа история коммитов сама по себе доказывает систематическую работу — это видно в GitHub-профиле.

## Недельный ритм

- **Дни 1–5** — новый материал по схеме выше
- **День 6** — только повторение и практика: прогоняешь ВСЕ карточки недели, переделываешь кейсы, дописываешь артефакт недели. Нового не берёшь
- **День 7** — выходной. Не героизм, а условие: без отдыха неделя 4 накроется

## Где брать первоисточники (не только чат)

**Процесс и методология:**

- **NIST SP 800-61** — Computer Security Incident Handling Guide. Библия incident response, читается тяжело, но это основа. _(проверь актуальную ревизию на сайте NIST)_
- **SANS Incident Handler's Handbook** — короче и практичнее NIST
- **Lockheed Martin Cyber Kill Chain** — оригинальный whitepaper, читается за час

**Практика (интерактив):**

- **TryHackMe** — модули для начинающего SOC-аналитика, SOC Level 1 path
- **LetsDefend** — симуляция очереди алертов, ближе всего к реальной работе L1
- **CyberDefenders** — кейсы уровня расследования

**Windows/AD:**

- Документация Microsoft по событиям безопасности (описание каждого Event ID)
- Ultimate Windows Security — справочник по Event ID с расшифровкой полей
- Sysmon: документация Sysinternals + конфиг SwiftOnSecurity с комментариями

**Атаки и детект:**

- **MITRE ATT&CK** — сама матрица + описания техник (там же секции Detection)
- **Atomic Red Team** — репозиторий тестов
- **Sigma** — репозиторий правил (читать чужие правила = лучший способ научиться писать свои)

**Сеть:**

- **Malware-Traffic-Analysis.net** — реальные PCAP с заданиями и ответами

**Русскоязычное:**

- Wazuh docs (англ., но основной инструмент)
- Anti-Malware.ru — обзоры отечественных SIEM (KUMA, MaxPatrol) для понимания рынка
- Хабр: разборы инцидентов от Solar, PT, BI.ZONE

## Как понять, что день прошёл не зря

Три критерия. Все три, не один:

1. **Есть артефакт** — файл в репозитории, которого утром не было
2. **Прошёл самопроверку** — ответил на вопросы дня, не подглядывая
3. **Объяснил вслух** — проговорил тему так, что понял бы человек не из ИБ

Если хоть один не выполнен — тема не закрыта, доносишь завтра в блок разгона.
